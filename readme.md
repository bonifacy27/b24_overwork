# Процессы по заявкам на сверхурочную работу

Проект содержит набор страниц и служебных скриптов Bitrix24 для создания, согласования, просмотра, редактирования и отмены заявок на сверхурочную работу, работу в выходной день и дежурство.

## Основные сущности

- **Заявка** — элемент инфоблока `IBLOCK_REQUESTS` (`391`). В заявке хранятся сотрудник, период работ, тип работ, тип оплаты, статус, история, расчет часов, связанные заявки и ссылка на группу.
- **Группа заявок** — элемент инфоблока `IBLOCK_GROUP_REQUESTS` (`397`). Используется при массовом создании заявок и связывает несколько заявок через свойство `GROUP_LINK` у заявки и `SVYAZANNYE_ZAYAVKI_GRUPPY` у группы.
- **Связанные заявки** — заявки, которые должны согласовываться или отклоняться синхронно через свойство `SVYAZANNYE_ZAYAVKI`.
- **Статусы** — элементы инфоблока статусов `IBLOCK_STATUSES` (`388`), включая статусы отмены, отклонения и переноса.

## Создание заявок

Страница `cr.php` отвечает за форму создания и AJAX-предпросмотр. Общая бизнес-логика вынесена в `inc/logic.php`, настройки инфоблоков и свойств — в `inc/constants.php`, данные для формы — в `inc/data.php`, интерфейс формы — в `inc/ui_form.php`.

Поддерживаются три режима создания:

1. **Одиночная заявка** (`single`) — создается заявка на одного сотрудника без группы.
2. **Несколько сотрудников с одинаковым периодом** (`multi_same`) — для выбранных сотрудников создаются заявки с общими датами и временем. Перед созданием заявок создается групповая заявка, а все созданные заявки получают ссылку на нее.
3. **Несколько сотрудников с разными периодами** (`multi_diff`) — для каждого сотрудника указывается собственный период. Заявки также объединяются общей групповой заявкой.

При создании выполняются проверки:

- доступ текущего пользователя к созданию заявок за выбранного сотрудника;
- запрет прошедших дат, если у пользователя нет специального доступа;
- обязательность типа оплаты;
- обязательность текстового обоснования для сверхурочной работы и работы в выходной день;
- предупреждение о подаче заявки менее чем за два рабочих дня;
- поиск пересечений с существующими заявками сотрудника;
- расчет сегментов по рабочим и выходным дням;
- расчет часов к оплате и расчетных HTML-таблиц для сверхурочных сегментов.

После успешного создания для каждой заявки запускается бизнес-процесс по шаблону `REQUEST_WORKFLOW_TEMPLATE_ID`.

## Групповые заявки

Групповая заявка создается при массовом режиме. Процесс состоит из двух связей:

- у каждой заявки заполняется свойство `GROUP_LINK` с ID групповой заявки;
- у групповой заявки заполняется свойство `SVYAZANNYE_ZAYAVKI_GRUPPY` со списком ID созданных заявок.

Группы используются в списке, карточке заявки и автосогласовании/автоотклонении групповых заявок.

## Просмотр и список заявок

- `list.php` выводит список заявок, фильтры, группировку связанных и групповых заявок, ссылки на карточку, задачи бизнес-процесса, отмену и редактирование. Список загружает все доступные заявки для применения поиска и фильтров по всему реестру, а затем разбивает результат на страницы по 40 записей.
- `view.php`, `view_full.php` и `view_simple.php` показывают карточку заявки, основные поля, историю, статус, расчет и соседние заявки той же группы.

## Согласование связанных заявок

Для синхронизации связанных заявок используются скрипты бизнес-процессов:

- `linked_approve.php` и CLI-вариант `linked_approve_cli.php` автоматически согласуют связанные заявки, если они находятся в нужном статусе и по ним есть активное задание БП.
- `linked_decline.php` автоматически отклоняет связанные заявки по аналогичной логике.

Скрипты пишут человекочитаемые строки в историю заявок и используют служебные маркеры, чтобы не выполнять одно и то же действие повторно.

## Согласование и отклонение групповых заявок

- `group_approve.php` обрабатывает автосогласование заявок из одной группы по свойству `GROUP_LINK`.
- `group_decline.php` обрабатывает автоотклонение заявок из одной группы.

Оба процесса находят все заявки группы, проверяют активные задания бизнес-процесса, выполняют действие от имени ответственного пользователя или администратора и добавляют записи в историю.

## Уточнение заявки

`refine.php` используется для возврата заявки на уточнение. Скрипт ищет активную задачу бизнес-процесса по заявке и текущему пользователю, получает параметры задания, отображает форму комментария без отдельного поля «Комментарии к доработке» и выполняет соответствующее действие БП.

## Редактирование заявки кадровым администратором

`edit_ka.php` реализует перенос заявки на новый период:

1. кадровый администратор открывает существующую заявку;
2. выбирает новую дату начала работ и указывает комментарий;
3. дата окончания рассчитывается автоматически с сохранением длительности исходной заявки;
4. создается новая заявка с теми же сотрудником, типом оплаты, обоснованием и файлами;
5. старая заявка получает статус **«Перенесена»**;
6. бизнес-процессы старой заявки прерываются;
7. в историю старой и новой заявки добавляются записи о переносе;
8. новая заявка связывается со старой и ранее связанными заявками;
9. если исходная заявка входила в группу, новая заявка получает ту же группу, а список заявок в групповой заявке дополняется ID новой заявки.

## Отмена выполненной заявки

`cancel.php` позволяет отменить только заявку в статусе **«Выполнена»**. При отмене проверяется текущий статус, отображаются ключевые параметры заявки, после подтверждения статус меняется на статус отмены, а история пополняется записью об отмене.

## Справочники и настройки

Основные ID инфоблоков, свойств, типов работ, статусов и подтипов находятся в `inc/constants.php`. При изменении структуры Bitrix24 сначала нужно обновить этот файл, а затем проверить создание, список, карточку, редактирование и бизнес-процессы.

## Типовой жизненный цикл заявки

1. Руководитель или уполномоченный пользователь создает заявку через `cr.php`.
2. Система выполняет проверки, расчет часов и создает одну или несколько заявок.
3. При массовом создании заявки объединяются группой.
4. По каждой заявке запускается бизнес-процесс согласования.
5. Согласующие выполняют задания БП; связанные и групповые заявки могут согласовываться или отклоняться автоматически.
6. Заявка переходит по статусам процесса до выполнения, отклонения, отмены или переноса.
7. Кадровый администратор при необходимости переносит заявку через `edit_ka.php`, создавая новую заявку вместо старой.
